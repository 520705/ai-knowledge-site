# 代码质量工具：ESLint / Prettier / Husky 实战指南

## 前言

代码质量是软件工程的核心要素之一。一套完善的代码质量工具链不仅能统一团队风格，更能提前发现潜在问题，将 Bug 消灭在萌芽阶段。本文档将深入讲解 ESLint、Prettier、Husky、lint-staged 与 commitlint 的完整配置与集成方案。

---

## 一、ESLint 完整配置

### 1.1 核心概念解析

ESLint 是 JavaScript/TypeScript 的静态代码分析工具，其工作原理如下：

```text
┌─────────────────────────────────────────────────────────────┐
│                      ESLint 工作流                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  源码 ──▶ 解析器(Parser) ──▶ 规则引擎 ──▶ 修复引擎 ──▶ 输出  │
│           │                        │                         │
│           ▼                        ▼                         │
│     AST 抽象语法树            报告问题/自动修复               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 安装与基础配置

```bash
# 安装 ESLint 及相关依赖
pnpm add -D eslint
pnpm add -D @typescript-eslint/parser @typescript-eslint/eslint-plugin
pnpm add -D eslint-plugin-react eslint-plugin-react-hooks
pnpm add -D eslint-plugin-jsx-a11y
pnpm add -D eslint-plugin-prettier eslint-config-prettier
```

### 3.2 安装与配置

```bash
# 安装 Husky
pnpm add -D husky lint-staged @commitlint/cli @commitlint/config-conventional

# 初始化 Husky（创建 .husky 目录）
npx husky init

# 创建 commit-msg hook
npx husky add .husky/commit-msg 'npx --no -- commitlint --edit ${1}'
```

### 3.3 lint-staged 配置

```json
// package.json
{
  "lint-staged": {
    "*.{js,jsx,ts,tsx}": [
      "prettier --write",
      "eslint --fix",
      "tsc --noEmit"
    ],
    "*.{json,css,md,html}": [
      "prettier --write"
    ],
    "*.{vue,svelte}": [
      "prettier --write",
      "eslint --fix"
    ]
  }
}
```

**工作原理**：

```bash
# 当你执行 git add src/App.tsx src/utils.ts
# lint-staged 只会对这两个文件运行检查
# 而不是整个项目的所有文件
```

### 3.4 commitlint 配置

```javascript
// commitlint.config.js
export default {
  extends: ['@commitlint/config-conventional'],
  rules: {
    'type-enum': [
      2,
      'always',
      [
        'feat',     // 新功能
        'fix',      // Bug 修复
        'docs',     // 文档变更
        'style',    // 代码格式（不影响功能）
        'refactor', // 重构（既不是新功能也不是修复）
        'perf',     // 性能优化
        'test',     // 添加/修改测试
        'build',    // 构建系统或依赖变更
        'ci',       // CI 配置变更
        'chore',    // 其他变更
        'revert',   // 回滚
        'wip',      // 开发中
      ],
    ],
    'type-case': [2, 'never', ['sentence-case', 'start-case', 'pascal-case', 'upper-case']],
    'type-empty': [2, 'never'],
    'subject-empty': [2, 'never'],
    'subject-full-stop': [2, 'never', '.'],
    'header-max-length': [2, 'always', 100],
  },
};
```

## 四、Git Hooks 手动管理

### 4.1 pre-push hook

```bash
# 创建 pre-push hook
npx husky add .husky/pre-push 'npx turbo run build --filter=./apps'
```

```bash
#!/usr/bin/env sh
. "$(dirname -- "$0")/_/husky.sh"

echo "Running pre-push checks..."

# 运行类型检查
echo "Type checking..."
npx tsc --noEmit || {
  echo "Type check failed!"
  exit 1
}

# 运行测试
echo "Running tests..."
pnpm test || {
  echo "Tests failed!"
  exit 1
}

echo "Pre-push checks passed!"
```

### 4.2 post-checkout hook

```bash
npx husky add .husky/post-checkout '#!/usr/bin/env sh
. "$(dirname -- "$0")/_/husky.sh"

# 安装依赖
pnpm install'
```

## 技术概述与定位

### 代码质量工具链的战略价值

在现代前端工程化体系中，代码质量工具链是保障代码健康、提高团队协作效率的关键基础设施。它不仅仅是一堆 linting 规则的集合，更是一套系统性的质量保障体系。

### 工具职责划分

| 工具 | 职责 | 检测类型 | 自动修复 |
|------|------|---------|---------|
| **Prettier** | 代码格式化 | 风格一致性 | 支持 |
| **ESLint** | 代码检查 | 代码质量、安全 | 部分支持 |
| **TypeScript** | 类型检查 | 类型错误 | 不支持 |
| **Husky** | Git Hooks | 流程控制 | 不支持 |
| **lint-staged** | 增量检查 | 性能优化 | 依赖下游工具 |
| **commitlint** | 提交规范 | 提交信息 | 不支持 |
| **Vitest** | 单元测试 | 功能正确性 | 不支持 |

## 核心概念详解

### 核心概念一：ESLint 工作原理

#### AST 抽象语法树

ESLint 工作的核心是理解代码的抽象语法树结构：

```
┌─────────────────────────────────────────────────────────────────────┐
│                    ESLint AST 解析流程                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  源代码                                                             │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │ const sum = (a, b) => {                                     │   │
│  │   return a + b;                                              │   │
│  │ };                                                           │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                            ▼                                        │
│                      词法分析                                        │
│                            ▼                                        │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  [TOKEN] const, identifier, =, (, identifier, ,, identifier, │   │
│  │        ), =>, {, return, identifier, +, identifier, ;, }      │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                            ▼                                        │
│                      语法分析                                        │
│                            ▼                                        │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  Program                                                     │   │
│  │  └── VariableDeclaration                                    │   │
│  │      └── VariableDeclarator                                 │   │
│  │          ├── id: Identifier (sum)                          │   │
│  │          └── init: ArrowFunctionExpression                  │   │
│  │              ├── params: [a, b]                            │   │
│  │              └── body: BlockStatement                       │   │
│  │                  └── body: ReturnStatement                  │   │
│  │                      └── argument: BinaryExpression (+, a, b) │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                            ▼                                        │
│                      规则检查                                        │
│                            ▼                                        │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  例如: no-unused-vars 规则检查是否有未使用的变量               │   │
│  │       eqeqeq 规则检查是否使用 === 而非 ==                     │   │
│  │       prettier/prettier 规则检查代码格式                       │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

#### 常见 AST 节点类型

| 节点类型 | 说明 | 示例 |
|---------|------|------|
| `Program` | 程序根节点 | 整个文件 |
| `VariableDeclaration` | 变量声明 | `const a = 1;` |
| `FunctionDeclaration` | 函数声明 | `function foo() {}` |
| `ArrowFunctionExpression` | 箭头函数 | `() => {}` |
| `CallExpression` | 函数调用 | `foo()` |
| `MemberExpression` | 成员访问 | `obj.prop` |
| `BinaryExpression` | 二元运算 | `a + b` |
| `UnaryExpression` | 一元运算 | `!a`, `-b` |
| `ConditionalExpression` | 条件表达式 | `a ? b : c` |
| `ArrayExpression` | 数组 | `[1, 2, 3]` |
| `ObjectExpression` | 对象 | `{ a: 1 }` |
| `Literal` | 字面量 | `1`, `"str"` |
| `Identifier` | 标识符 | 变量名 |

### 核心概念二：lint-staged 工作原理

#### 为什么需要 lint-staged

传统 pre-commit hook 的问题：

```
┌─────────────────────────────────────────────────────────────────────┐
│                   传统 pre-commit vs lint-staged                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  传统 pre-commit:                                                   │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │ git add file1.ts file2.ts                                   │   │
│  │ git commit -m "fix: resolve bug"                             │   │
│  │                                                              │   │
│  │ → 运行 ESLint 检查全部文件（可能 1000+ 个）                    │   │
│  │ → 耗时 30 秒+                                                 │   │
│  │ → 可能因为其他文件的错误而失败                                  │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│  lint-staged:                                                       │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │ git add file1.ts file2.ts                                   │   │
│  │ git commit -m "fix: resolve bug"                             │   │
│  │                                                              │   │
│  │ → 只检查已暂存的文件（2 个文件）                                │   │
```

**2. commit-msg Hook：自定义规则**

```javascript
// commit-msg.js
const fs = require('fs');

const commitTypes = {
  feat: '新功能',
  fix: 'Bug 修复',
  docs: '文档变更',
  style: '代码格式',
  refactor: '重构',
  perf: '性能优化',
  test: '测试',
  build: '构建',
  ci: 'CI',
  chore: '其他',
  revert: '回滚',
};

const msg = fs.readFileSync(process.argv[2], 'utf8').trim();
const pattern = /^(feat|fix|docs|style|refactor|perf|test|build|ci|chore|revert)(\(.+\))?: .+/;

if (!pattern.test(msg)) {
  console.error(`
❌ 无效的提交信息格式！

允许的格式: <type>(<scope>): <subject>

类型:
${Object.entries(commitTypes)
  .map(([key, value]) => `  - ${key}: ${value}`)
  .join('\n')}

示例:
  feat(auth): 添加登录功能
  fix(ui): 修复按钮样式问题
  docs: 更新 README
`);
  process.exit(1);
}

console.log('✅ 提交信息格式正确！');
```

## 常见问题与解决方案

### 问题一：ESLint 和 Prettier 冲突

**症状：**

ESLint 修复后，Prettier 又把格式改回来，形成循环。

**解决方案：**

1. **安装 eslint-config-prettier**

```bash
pnpm add -D eslint-config-prettier
```

2. **在 ESLint 配置中禁用冲突规则**

```javascript
// eslint.config.js
import prettierConfig from 'eslint-config-prettier';

export default [
  // ... 其他配置
  
  // Prettier 配置（必须放在最后）
  {
    plugins: {
      prettier: prettier,
    },
    rules: {
      ...prettierConfig.rules,
      'prettier/prettier': 'error',
    },
  },
];
```

### 问题二：commitlint 阻止合法提交

**症状：**

使用了正确的格式但仍被阻止。

**解决方案：**

1. **检查配置文件是否正确加载**

```bash
# 验证配置
npx commitlint --verbose
```

2. **检查 git hook 是否正确设置**

```bash
cat .husky/commit-msg
# 应该包含: npx --no -- commitlint --edit ${1}
```

3. **临时跳过 hook（紧急情况）**

```bash
git commit --no-verify -m "emergency fix"
```

> [!WARNING]
> 谨慎使用 `--no-verify`，仅在紧急情况下使用。

### 问题三：lint-staged 太慢

**症状：**

每次提交都要等很久。

**解决方案：**

1. **使用缓存**

```json
{
  "lint-staged": {
    "*.ts": [
      "eslint --cache --fix",
      "tsc --noEmit --incremental"
    ]
  }
}
```

2. **并行执行**

```javascript
// lint-staged.config.js
module.exports = {
  '*.{ts,tsx}': async (files) => {
    const filesArray = files.split(' ');
    // 并行执行多个任务
    await Promise.all([
      exec(`prettier --write ${filesArray.join(' ')}`),
      exec(`eslint --fix ${filesArray.join(' ')}`),
    ]);
  },
};
```

3. **减少检查范围**

```json
{
  "lint-staged": {
    "*.ts": "eslint --fix --cache --max-warnings 0"
  }
}
```

### 问题四：ESLint 在大型 Monorepo 中性能差

**症状：**

ESLint 检查整个仓库太慢。

**解决方案：**

1. **使用 Workspace 过滤**

```bash
# 只检查变更的包
pnpm --filter "@my/web" lint
```

2. **配置缓存**

```javascript
// eslint.config.js
export default {
  // ... 其他配置
  
  // 启用缓存
  cache: true,
  cacheLocation: './node_modules/.cache/eslint/',
};
```

3. **分离类型检查**

```javascript
// 不在 ESLint 中进行类型检查，使用独立的 tsc --noEmit
```

### 问题五：Git Hooks 不生效

**症状：**

push 后 CI 检查失败，但 pre-push hook 没有拦截。

**解决方案：**

1. **检查 Husky 版本**

```bash
npx husky --version
# 应该 >= 9.0.0
```

2. **检查 hook 文件权限**

```bash
ls -la .husky/pre-push
# 应该有执行权限 (-rwxr-xr-x)
chmod +x .husky/pre-push
```

3. **重新初始化**

```bash
rm -rf .husky
npm uninstall husky
npm install -D husky
npx husky init
```

### 示例二：自动化发布工作流

```yaml
# .github/workflows/release.yml
name: Release

on:
  push:
    tags:
      - 'v*'

jobs:
  release:
    name: Release
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'pnpm'

      - run: pnpm install --frozen-lockfile

      - name: Lint
        run: pnpm lint

      - name: Typecheck
        run: pnpm typecheck

      - name: Test
        run: pnpm test:coverage

      - name: Build
        run: pnpm build

      - name: Publish
        env:
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
        run: |
          pnpm publish --access public --no-git-checks

      - name: Create GitHub Release
        uses: softprops/action-gh-release@v1
        with:
          files: dist/*
          generate_release_notes: true
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```
