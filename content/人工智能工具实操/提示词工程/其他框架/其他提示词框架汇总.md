# 其他提示词框架汇总 - 找到适合你的那一款

> **这篇文章解决什么问题**：COSTAR太复杂？有没有更简单的框架？这篇文章介绍ICIO、RTF、APE、SOP等多个框架，总有一款适合你。
> 
> **你需要什么基础**：零基础即可
> 
> **看完能做什么**：了解多种框架，根据不同场景选择最合适的
> 
> **更新日期**：2026年4月

## 一、先说清楚：什么是"提示词框架"？

### 1.1 框架是什么？

提示词框架就是帮你组织提示词结构的"模板"。

就像你写文章有开头、中间、结尾的结构，写提示词也有结构可循。框架就是告诉你：想要AI给出好答案，你需要说清楚哪几件事。

### 1.2 为什么要用框架？

好处有三个：

1. **不遗漏**：照着框架过一遍，该说的都说了
2. **更规范**：结构清晰，AI更容易理解
3. **可迭代**：框架定了，调试和优化更容易

### 1.3 这篇文章介绍哪些框架？

| 框架 | 组件数 | 难度 | 最适合场景 |
|------|--------|------|------------|
| ICIO | 4个 | ⭐ | 快速简单任务 |
| RTF | 3个 | ⭐ | 需要特定格式 |
| APE | 3个 | ⭐⭐ | 规划行动 |
| SOP | N个 | ⭐⭐ | 标准化流程 |
| SxS | 3个 | ⭐ | 超快速任务 |

## 二、ICIO框架 - 快速简单任务用

### 2.1 ICIO是什么？

ICIO是最简洁的框架之一，只有4个要素：

| 字母 | 含义 | 你要回答 |
|------|------|----------|
| **I** | Instruction | 要做什么？ |
| **C** | Context | 什么背景？ |
| **I** | Input | 输入什么？ |
| **O** | Output | 输出什么格式？ |

### 2.2 什么时候用ICIO？

**适合场景**：
- 快速翻译
- 简单润色
- 格式转换
- 短文本处理

**不适合**：
- 复杂推理任务
- 需要角色扮演
- 高质量内容创作

### 2.3 ICIO模板

```markdown
# Instruction（指令）
你要做什么？

# Context（背景）
相关的背景信息是什么？

# Input（输入）
需要处理的具体内容是什么？

# Output（输出）
期望的输出格式是什么样的？
```

### 2.4 ICIO实战例子

**例子：翻译任务**

```
# Instruction
将以下英文文本翻译成中文，保持专业术语的一致性。

# Context
这是一份医疗器械的用户手册，需要在严谨性和可读性之间取得平衡。
目标读者是医院护士，非医学专业背景。

# Input
"The device should be operated by qualified personnel only. 
Before use, ensure the battery is fully charged. 
In case of malfunction, contact technical support immediately."

# Output
- 医学术语采用国家标准译名
- 句子结构符合中文习惯
- 可以添加适当注释帮助理解
```

**例子：润色文案**

```
# Instruction
帮我润色以下文案，让它更有感染力。

# Context
这是我们品牌的一篇推广文案，目标读者是25-35岁的都市女性，
她们关注自我成长，追求品质生活。

# Input
"我们的产品可以帮助你提升效率。
采用了最新技术，质量很好。
价格合理，欢迎购买。"

# Output
- 保留原意，增加感染力
- 口语化但不失专业
- 长度控制在150字以内
- 避免过度营销化
```

## 三、RTF框架 - 需要特定格式用

### 3.1 RTF是什么？

RTF专注于格式控制，只有3个要素：

| 字母 | 含义 | 你要回答 |
|------|------|----------|
| **R** | Role | AI扮演什么角色？ |
| **T** | Task | 需要完成什么任务？ |
| **F** | Format | 输出格式是什么？ |

### 3.2 什么时候用RTF？

**适合场景**：
- 数据提取
- API响应格式化
- 结构化输出
- 批量处理

**不适合**：
- 创意写作
- 复杂分析
- 需要风格控制的场景

### 3.3 RTF模板

```markdown
# Role
你是什么角色？

# Task
需要完成什么具体任务？

# Format
输出格式是什么？
- 格式类型（JSON/CSV/表格...）
- 必须包含的字段
- 结构要求
```

### 3.4 RTF实战例子

**例子：提取信息**

```
# Role
你是一位数据分析专家，擅长从非结构化文本中提取结构化信息。

# Task
从以下新闻文章中提取关键信息。

# Format
以JSON格式输出：
{
  "event": "事件描述",
  "persons": ["人物列表"],
  "date": "YYYY-MM-DD格式",
  "location": "地点",
  "impact": "影响描述"
}

---

输入文本：
"2026年4月15日，特斯拉CEO马斯克宣布将在上海建立新的研发中心，
预计投资50亿美元，将创造10000个就业岗位。这一消息得到了
上海市政府的确认和欢迎。"
```

**例子：生成测试数据**

```
# Role
你是一位数据生成专家。

# Task
生成5条模拟用户评价数据。

# Format
以JSON数组格式输出：
[
  {
    "user_id": "用户ID",
    "rating": "1-5的评分",
    "comment": "评价内容",
    "date": "YYYY-MM-DD"
  }
]

要求：
- 用户ID格式：user_001
- 评分分布：3条5星，2条4星
- 内容要真实可信
```

## 四、APE框架 - 规划行动用

### 4.1 APE是什么？

APE专注于行动规划，有3个要素：

| 字母 | 含义 | 你要回答 |
|------|------|----------|
| **A** | Action | 具体做什么？ |
| **P** | Purpose | 为什么要做？ |
| **E** | Expectation | 期望什么结果？ |

### 4.2 什么时候用APE？

**适合场景**：
- 项目规划
- 战略制定
- 行动方案设计
- 流程优化

**不适合**：
- 直接回答问题
- 简单问答
- 创意写作

### 4.3 APE模板

```markdown
# Action
具体要做什么？
- 步骤1
- 步骤2
- 步骤3

# Purpose
为什么要做这些？
- 解决什么问题？
- 创造什么价值？
- 跟大目标的关联？

# Expectation
期望达成什么结果？
- 成功的标志是什么？
- 如何验证完成度？
- 有哪些质量要求？
```

### 4.4 APE实战例子

**例子：制定营销计划**

```
# Action
为下季度的产品发布制定营销计划。
步骤1：分析目标市场和竞争格局
步骤2：确定目标客户画像和核心信息
步骤3：选择渠道并制定预算分配
步骤4：制定内容日历和关键里程碑

# Purpose
- 目标：在产品发布后60天内获得500个付费用户
- 背景：产品定位于中小企业市场，面临3家主要竞争者
- 预算：营销预算为10万美元
- 大目标：助力公司完成A轮融资

# Expectation
交付物：
- 完整的营销计划文档
包含：市场分析、策略选择、预算分配、时间表
格式：可执行级别的详细计划
```

**例子：设计学习计划**

```
# Action
帮我设计一个3个月的产品经理学习计划。
步骤1：确定学习目标和资源
步骤2：制定每周学习内容
步骤3：设计实践项目
步骤4：设置检验节点

# Purpose
- 目标：3个月后能够独立负责一个小产品
- 背景：我有1年运营经验，转岗产品经理
- 约束：每天只能学习2小时，周末4小时
- 动机：产品经理薪资更高，发展空间更大

# Expectation
- 阶段成果：每月一个阶段性产出
- 最终成果：完成一个完整的项目作品集
- 验收方式：能回答面试中的产品问题
```

## 五、SOP框架 - 标准化流程用

### 5.1 SOP是什么？

SOP是"标准操作流程"的缩写，专注于流程设计：

| 要素 | 含义 |
|------|------|
| **S**cope | 流程的范围 |
| **O**bjective | 流程的目标 |
| **P**rocess | 具体步骤 |

### 5.2 什么时候用SOP？

**适合场景**：
- 流程文档编写
- 培训材料制作
- 操作手册生成
- 合规文档

### 5.3 SOP模板

```markdown
# Scope（范围）
流程的适用范围
- 适用场景
- 不适用场景
- 前置条件

# Objective（目标）
流程要达成的主要目标
- 主要目标
- 次要目标

# Process（流程）
## 步骤1：[名称]
### 操作说明
[详细操作步骤]

### 检查点
[该步骤的质量检查项]

## 步骤2：[名称]
...

# Quality（质量）
- 完成标准
- 常见问题处理
- 记录要求
```

### 5.4 SOP实战例子

**例子：客服处理流程**

```
# Scope
适用范围：新用户投诉处理
适用部门：客户成功团队
前置条件：已确认客户身份和投诉内容

# Objective
主要目标：在24小时内回复，72小时内解决问题
次要目标：收集投诉数据用于产品改进

# Process

## 步骤1：接收与记录
操作说明：
1. 在系统中创建投诉工单
2. 记录客户信息、投诉内容和期望
3. 评估紧急程度（P1/P2/P3）
4. 分配给相应处理人

检查点：
- 信息完整度100%
- 紧急程度评估合理

## 步骤2：调查与分析
操作说明：
1. 联系相关团队了解情况
2. 收集相关证据和日志
3. 分析问题根因
4. 制定解决方案

检查点：
- 问题根因明确
- 解决方案可行

## 步骤3：回复与跟进
操作说明：
1. 回复客户，说明进展
2. 执行解决方案
3. 确认客户满意
4. 关闭工单

检查点：
- 回复时间符合SLA
- 客户确认满意

# Quality
- 响应时间：P1=4小时，P2=12小时，P3=24小时
- 解决率目标：90%
- 客户满意度：4.5/5.0
```

## 六、SxS框架 - 超快速任务用

### 6.1 SxS是什么？

SxS是最简单的框架，只有3个要素：

| 字母 | 含义 |
|------|------|
| **S**ubject | 主题 |
| **S**ituation | 情况 |
| **S**tatement | 任务 |

### 6.2 什么时候用SxS？

**适合场景**：
- 超快速任务描述
- 简单沟通
- 头脑风暴启动
- 快速邮件草稿

### 6.3 SxS模板

```
Subject: [主题]
Situation: [当前情况]
Statement: [需要完成的任务]
```

### 6.4 SxS实战例子

```
Subject: 季度销售报告
Situation: 第三季度结束，我们需要向董事会汇报销售业绩
Statement: 撰写一份包含关键指标、达成情况和下季度预测的销售报告
```

```
Subject: 产品功能反馈
Situation: 用户反馈我们的App启动速度太慢，给了1星评价
Statement: 分析启动速度慢的可能原因，并给出优化建议
```

```
Subject: 面试准备
Situation: 明天有产品经理岗位的面试，对方是一家B轮融资的SaaS公司
Statement: 帮我准备可能被问到的10个高频问题及回答思路
```

## 七、框架对比与选择

### 7.1 一图总结

| 框架 | 复杂度 | 最适合 | 最不适合 |
|------|--------|--------|----------|
| COSTAR | ⭐⭐⭐ | 复杂高质量任务 | 简单快速任务 |
| ICIO | ⭐ | 快速简单任务 | 复杂任务 |
| RTF | ⭐ | 格式控制 | 创意写作 |
| APE | ⭐⭐ | 规划行动 | 直接回答 |
| SOP | ⭐⭐ | 流程文档 | 开放式任务 |
| SxS | ⭐ | 超快速任务 | 任何需要细节的任务 |

### 7.2 选择决策树

```
任务类型 → 推荐框架
│
├─ 需要快速搞定 → SxS
│
├─ 简单翻译/润色 → ICIO
│
├─ 需要特定格式 → RTF
│
├─ 需要规划行动 → APE
│
├─ 需要写流程文档 → SOP
│
└─ 复杂高质量任务 → COSTAR
```

### 7.3 可以组合使用

框架不是非此即彼，可以组合：

## 八、实战建议

### 8.1 新手建议

**第一步**：从ICIO或SxS开始
- 简单好上手
- 能处理日常大部分任务

**第二步**：学RTF
- 当你需要特定格式输出时
- 比如提取信息、生成结构化数据

**第三步**：学APE或SOP
- 当你需要规划或流程时
- 比如制定计划、写操作手册

**第四步**：学COSTAR
- 当你需要高质量复杂任务时
- 需要全面控制输出

### 8.2 记住的原则

1. **框架是工具，不是枷锁**：不要为了用框架而用框架
2. **简单任务简单处理**：一个SxS能搞定的，不用COSTAR
3. **根据需要调整**：框架不是死的，可以删减组合
4. **最重要的是说清楚**：不管用什么框架，核心是把话说清楚

## 九、总结

| 框架 | 一句话说明 | 使用场景 |
|------|-----------|----------|
| ICIO | 简单四要素 | 翻译、润色、快速问答 |
| RTF | 角色+任务+格式 | 数据提取、格式化输出 |
| APE | 行动+目的+期望 | 规划、战略、方案设计 |
| SOP | 范围+目标+流程 | 流程文档、操作手册 |
| SxS | 主题+情况+任务 | 超快速任务、头脑风暴 |

**核心建议**：
- 不要死记硬背框架
- 理解每个框架解决的问题
- 根据任务需要灵活选择
- 框架可以组合使用
