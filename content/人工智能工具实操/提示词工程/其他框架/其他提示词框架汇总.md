# 其他提示词框架汇总

> **关键词**：ICIO、RTF、APE、SOP、TEMPLATE、SxS、SCALE、框架对比、结构化提示词、框架选择

## 一、框架概览与分类

在提示词工程领域，除了广为人知的COSTAR、CRISPE、BROKE等框架外，还存在着一系列针对特定场景设计的提示词框架。这些框架各有其独特的优势和适用场景，本文档将对ICIO、RTF、APE、SOP、TEMPLATE、SxS和SCALE七个框架进行系统性介绍和对比分析。

> [!note] 框架选择原则
> 没有"最佳"框架，只有"最适合"的框架。选择框架时应考虑：任务复杂度、时间约束、输出质量要求、交互方式等因素。

## 二、ICIO框架

### 2.1 框架概述

ICIO框架是由AI研究者提出的一个极简主义框架，适用于快速、简单的任务场景。框架名称来源于四个核心要素：**I**nstruction（指令）、**C**ontext（上下文）、**I**nput Data（输入数据）、**O**utput Indicator（输出指示）。

### 2.2 组件详解

| 组件 | 核心问题 | 必填性 |
|------|----------|--------|
| Instruction | 需要AI做什么？ | 必须 |
| Context | 相关背景信息是什么？ | 建议 |
| Input Data | 需要处理什么数据？ | 视情况 |
| Output Indicator | 期望什么样的输出？ | 建议 |

### 2.3 模板与示例

```
# Instruction
[明确告诉AI需要执行的具体任务]

# Context
[提供必要的背景信息]

# Input Data
[需要处理的具体数据或内容]

# Output Indicator
[期望的输出格式、质量标准或约束]
```

**示例**：
```
# Instruction
将以下英文文本翻译成中文，保持专业术语的一致性。

# Context
这是一份医疗器械的用户手册，需要在严谨性和可读性之间取得平衡。

# Input Data
"The device should be operated by qualified personnel only..."

# Output Indicator
- 医学术语采用国家标准译名
- 句子结构符合中文习惯
- 难度等级：普通用户可理解
```

### 2.4 适用场景与局限性

**最佳场景**：
- 快速翻译任务
- 简单的格式转换
- 短文本处理
- 一次性查询

**局限性**：
- 不适合复杂推理任务
- 缺乏角色和风格控制
- 难以保证输出的一致性

## 三、RTF框架

### 3.1 框架概述

RTF框架专注于**格式控制**，适用于需要特定输出结构的场景。框架名称来源于三个核心要素：**R**ole（角色）、**T**ask（任务）、**F**ormat（格式）。与COSTAR相比，RTF将表达风格的控制简化为格式约束，更适合结构化数据输出。

### 3.2 组件详解

| 组件 | 核心问题 | 描述 |
|------|----------|------|
| Role | AI扮演什么角色？ | 定义专业身份和能力边界 |
| Task | 需要完成什么任务？ | 具体、明确的任务描述 |
| Format | 输出格式是什么？ | 结构化要求、模板约束 |

### 3.3 模板与示例

```
# Role
[定义AI的专业角色和知识领域]

# Task
[具体要执行的任务描述]

# Format
[期望的输出格式要求]
- 格式类型（JSON/CSV/Markdown等）
- 必须包含的字段
- 结构层次
- 示例输出
```

**示例**：
```
# Role
你是一位数据分析专家，擅长从非结构化文本中提取结构化信息。

# Task
从以下新闻文章中提取关键信息：事件、涉及的人物、时间、地点、影响。

# Format
以JSON格式输出，包含以下字段：
{
  "event": "事件描述",
  "persons": ["人物列表"],
  "date": "YYYY-MM-DD格式",
  "location": "地点",
  "impact": "影响描述"
}
```

### 3.4 适用场景与局限性

**最佳场景**：
- 数据提取任务
- API响应格式化
- 数据库记录生成
- 表单填写

**局限性**：
- 不适合创意写作
- 缺乏风格控制
- 上下文依赖性高

## 四、APE框架

### 4.1 框架概述

APE框架专注于**行动规划**，适用于需要分解步骤或规划行动方案的场景。框架名称来源于三个核心要素：**A**ction（行动）、**P**urpose（目的）、**E**xpectation（期望）。APE特别强调"为什么做"，而非仅仅"做什么"。

### 4.2 组件详解

| 组件 | 核心问题 | 描述 |
|------|----------|------|
| Action | 具体要做什么？ | 可执行的行动或任务 |
| Purpose | 为什么要做？ | 行动背后的目的和动机 |
| Expectation | 期望达成什么结果？ | 成功标准和验收条件 |

### 4.3 模板与示例

```
# Action
[需要执行的具体行动或任务]
步骤1: [子步骤]
步骤2: [子步骤]
步骤3: [子步骤]

# Purpose
[行动的目的和背景]
- 解决什么问题？
- 创造什么价值？
- 与更大目标的关联？

# Expectation
[期望的结果和验收标准]
- 成功的标志是什么？
- 如何验证完成度？
- 有哪些质量要求？
```

**示例**：
```
# Action
为下季度的产品发布制定营销计划。
步骤1: 分析目标市场和竞争格局
步骤2: 确定目标客户画像和核心信息
步骤3: 选择渠道并制定预算分配
步骤4: 制定内容日历和关键里程碑

# Purpose
- 目标：在产品发布后60天内获得500个付费用户
- 背景：产品定位于中小企业市场，面临3家主要竞争者
- 预算：营销预算为10万美元

# Expectation
- 交付物：完整的营销计划文档
- 包含：市场分析、策略选择、预算分配、时间表
- 格式：可执行级别的详细计划
```

### 4.4 适用场景与局限性

**最佳场景**：
- 项目规划
- 战略制定
- 行动方案设计
- 流程优化

**局限性**：
- 不适合需要直接回答的问题
- 缺乏详细的风格控制
- 模板较为固定

## 五、SOP框架

### 5.1 框架概述

SOP框架专为**标准操作流程**设计，适用于需要生成可执行、可复制流程的场景。框架名称直接来自Standard Operating Procedure（标准操作流程）的缩写。SOP框架强调步骤的完整性、可重复性和可验证性。

### 5.2 组件详解

| 组件 | 核心问题 | 描述 |
|------|----------|------|
| Scope | 流程的范围是什么？ | 边界定义和适用范围 |
| Objective | 流程的目标是什么？ | 最终要达成什么 |
| Process | 具体步骤是什么？ | 详细的操作步骤 |
| Quality | 质量标准是什么？ | 验收标准和检查点 |

### 5.3 模板与示例

```
# Scope
[流程的适用范围和边界]
- 适用场景
- 不适用场景
- 前置条件

# Objective
[流程要达成的主要目标]
- 主要目标
- 次要目标（如有）

# Process
## 步骤1: [步骤名称]
### 操作说明
[详细操作步骤]

### 检查点
[该步骤的质量检查项]

## 步骤2: [步骤名称]
...

# Quality
[质量标准和验收条件]
- 完成标准
- 常见问题处理
- 记录和文档要求
```

**示例**：
```
# Scope
适用范围：客户投诉处理
适用部门：客户成功团队
前置条件：已确认客户身份和投诉内容

# Objective
主要目标：在24小时内回复客户，72小时内解决问题
次要目标：收集投诉数据用于产品改进

# Process
## 步骤1: 投诉接收与记录
### 操作说明
1. 在CRM系统中创建投诉工单
2. 记录客户基本信息、投诉内容和期望
3. 评估紧急程度（P1/P2/P3）
4. 分配给相应处理人

### 检查点
□ 信息完整度100%
□ 紧急程度评估合理

## 步骤2: 调查与分析
...

# Quality
- 响应时间：P1=4小时，P2=12小时，P3=24小时
- 解决率目标：90%
- 客户满意度目标：4.5/5.0
```

### 5.4 适用场景与局限性

**最佳场景**：
- 流程文档编写
- 培训材料制作
- 操作手册生成
- 合规文档

**局限性**：
- 不适合创意性任务
- 模板较为刚性
- 需要领域专业知识

## 六、TEMPLATE框架

### 6.1 框架概述

TEMPLATE框架使用**占位符填充**的方式，适用于需要批量生成相似内容的场景。框架通过预定义的模板结构，允许用户通过填充变量快速生成一致的输出。

### 6.2 组件详解

| 组件 | 核心问题 | 描述 |
|------|----------|------|
| Template | 输出模板是什么？ | 包含占位符的输出结构 |
| Variables | 需要填充哪些变量？ | 变量列表和说明 |
| Rules | 有什么规则约束？ | 格式、长度、质量要求 |

### 6.3 模板与示例

```
# Template
[包含占位符的输出模板]
{{变量1}}
{{变量2}}
...

# Variables
变量1: [变量1的说明和来源]
变量2: [变量2的说明和来源]
...

# Rules
[填充时的规则和约束]
- 格式要求
- 长度限制
- 质量标准
```

**示例**：
```
# Template
# 产品发布公告

**产品名称**: {{product_name}}
**发布日期**: {{release_date}}
**版本号**: {{version}}

## 新功能

{{feature_list}}

## 升级说明

{{upgrade_guide}}

## 获取方式

{{download_link}}
```

### 6.4 适用场景与局限性

**最佳场景**：
- 批量内容生成
- 标准化文档
- 邮件模板
- 报告生成

**局限性**：
- 不适合开放式创作
- 灵活性较低
- 依赖模板设计质量

## 七、SxS框架

### 7.1 框架概述

SxS（**S**ubject + **S**ituation + **S**tatement）框架是一个**极度简化**的框架，适用于快速单轮对话场景。它通过三个核心要素快速定位任务上下文。

### 7.2 组件详解

| 组件 | 核心问题 | 描述 |
|------|----------|------|
| Subject | 主题是什么？ | 讨论的对象 |
| Situation | 当前情况是什么？ | 场景背景 |
| Statement | 需要做什么/说什么？ | 具体行动或内容 |

### 7.3 模板与示例

```
Subject: [主题]
Situation: [当前情况]
Statement: [需要完成的任务或内容]
```

**示例**：
```
Subject: 季度销售报告
Situation: 第三季度结束，我们需要向董事会汇报销售业绩
Statement: 撰写一份包含关键指标、达成情况和下季度预测的销售报告
```

### 7.4 适用场景与局限性

**最佳场景**：
- 快速任务描述
- 简单沟通
- 头脑风暴启动
- 快速邮件草稿

**局限性**：
- 不适合复杂任务
- 缺乏细节控制
- 输出质量不稳定

## 八、SCALE框架

### 8.1 框架概述

SCALE框架专注于**大规模内容生成**场景，适用于需要保持一致性的批量任务。框架名称来源于五个核心要素：**S**ummarize（总结）、**C**larify（澄清）、**A**nalyze（分析）、**L**ogic（逻辑）、**E**valuate（评估）。

### 8.2 组件详解

| 组件 | 核心问题 | 描述 |
|------|----------|------|
| Summarize | 核心信息是什么？ | 内容摘要 |
| Clarify | 需要澄清什么？ | 关键问题澄清 |
| Analyze | 如何分析？ | 分析方法 |
| Logic | 逻辑是什么？ | 论证逻辑 |
| Evaluate | 如何评估？ | 质量评估标准 |

### 8.3 模板与示例

```
# Summarize
[待处理内容的核心摘要]

# Clarify
[需要澄清的关键问题]
Q1: [问题1]
Q2: [问题2]

# Analyze
[采用的分析方法]
- 数据来源
- 分析维度
- 工具/技术

# Logic
[论证逻辑结构]
[前提]
  → [推论1]
    → [结论1]
  → [推论2]
    → [结论2]

# Evaluate
[输出质量评估标准]
- 准确性要求
- 完整性要求
- 一致性要求
```

### 8.4 适用场景与局限性

**最佳场景**：
- 大规模文档分析
- 批量报告生成
- 研究综述
- 系统性内容审核

**局限性**：
- 框架相对复杂
- 不适合快速任务
- 需要专业分析能力

## 九、框架综合对比

### 9.1 功能维度对比

| 框架 | 组件数 | 复杂度 | 适用场景数 | 风格控制 | 格式控制 | 目标驱动 |
|------|--------|--------|------------|----------|----------|----------|
| ICIO | 4 | 低 | 少 | 弱 | 中 | 弱 |
| RTF | 3 | 低 | 中 | 弱 | 强 | 弱 |
| APE | 3 | 中 | 中 | 中 | 弱 | 强 |
| SOP | 4 | 高 | 少 | 弱 | 强 | 中 |
| TEMPLATE | 3 | 中 | 中 | 弱 | 强 | 弱 |
| SxS | 3 | 极低 | 少 | 弱 | 弱 | 弱 |
| SCALE | 5 | 高 | 中 | 中 | 中 | 强 |

### 9.2 选择决策树

```
任务类型 → 推荐框架
│
├─ 快速简单任务 → ICIO / SxS
│
├─ 需要特定格式 → RTF / TEMPLATE
│
├─ 需要规划行动 → APE / BROKE
│
├─ 需要完整流程 → SOP / SCALE
│
├─ 需要全面质量 → COSTAR / CRISPE
│
└─ 需要目标管理 → BROKE
```

### 9.3 框架组合策略

在实际应用中，可以组合多个框架以发挥各自优势：

1. **ICIO + RTF**：快速任务+格式要求
   ```
   # ICIO结构 + RTF格式要求
   # Instruction + Format
   ```

2. **APE + SCALE**：行动规划+系统性分析
   ```
   # APE行动框架 + SCALE分析框架
   # 先规划行动，再系统分析
   ```

3. **SOP + COSTAR**：流程文档+高质量表达
   ```
   # SOP流程结构 + COSTAR风格
   # 完整流程+专业表达
   ```

## 十、实践建议

### 10.1 初学者建议

> [!tip] 新手入门路径
> 1. 从极简框架开始：SxS或ICIO
> 2. 逐步增加复杂度：RTF或APE
> 3. 根据需要选择完整框架：COSTAR或BROKE
> 4. 尝试框架组合应用

### 10.2 框架定制建议

每个框架都不是一成不变的，可以根据个人需求进行定制：

1. **简化框架**：移除不必要的组件
2. **增强框架**：添加特定领域的组件
3. **组合框架**：混合多个框架的组件
4. **本土化框架**：根据中文语境调整

### 10.3 框架演进路径

建议根据使用经验逐步演进：

| 阶段 | 建议框架 | 学习重点 |
|------|----------|----------|
| 入门 | ICIO/SxS | 理解基本结构 |
| 进阶 | RTF/APE | 掌握格式和规划 |
| 熟练 | COSTAR/BROKE | 全面质量控制 |
| 专家 | 自定义框架 | 根据需求创新 |
