## 概述

提示词工程（Prompt Engineering）是与大型语言模型（LLM）交互的核心技能，涉及设计、优化和管理输入提示词以获得理想输出的全部技术。从 2020 年 GPT-3 的 few-shot learning，到 2025-2026 年的推理模型时代，提示词工程的内涵已从"措辞选择"演进为一套系统的工程学科。

现代提示词工程已超越单一框架的边界，形成了**结构化提示框架**（CO-STAR、CRISPE、ICIO、BROKE、SOP 等）、**推理增强技术**（CoT、ToT、ReAct、Self-Consistency）、**高级应用模式**（Agentic Prompting、RAG 提示词、多模态提示）以及**生产级最佳实践**的完整体系。本文档将逐一展开每个维度的原理、模板与实战案例。

## 一、CO-STAR 框架：竞赛冠军的结构化方法

### 1.1 框架起源与核心原理

CO-STAR 由数据科学家 Sheila Teo 提出，在新加坡首届 GPT-4 提示词工程竞赛中夺得冠军，现已被 Anthropic 官方工程师等专业人士广泛采用。它的核心设计哲学是**为 AI 提供充分且必要的上下文，使输出精准匹配预期**。

CO-STAR 之所以有效，是因为它系统性地解决了提示词工程中的六大核心维度：

- **减少幻觉**：通过具体上下文约束 AI 的生成空间
- **控制风格一致性**：显式分离 Style 和 Tone，避免两者混淆
- **降低成本与调试时间**：据实测可减少 60% 的 API 调用和调试工作量
- **增强目标导向**：明确 Objective 使 AI 聚焦于"做什么"而非"怎么做"

### 1.2 六个组件详解

#### C - Context（上下文）

提供任务背景、约束条件、项目结构和当前限制。上下文越具体，AI 越能理解具体场景。

```
# ❌ 弱上下文
帮我写一段代码。

# ✅ 强上下文
我在开发一个 React Native 移动应用，目前处于 MVP 阶段。
用户可以在应用中记录每日饮食。当前需要实现一个搜索功能，
允许用户通过食材名称查找已有饮食记录。请注意：
- 数据存储使用 SQLite
- 需要支持模糊匹配
- 移动端网络不稳定，需离线优先
```

#### O - Objective（目标）

清晰定义你希望 AI 完成的具体任务，聚焦于"什么"而非"怎么做"。

```
# ❌ 模糊目标
帮我优化这个算法。

# ✅ 清晰目标
分析以下函数的性能瓶颈，提供具体优化方案。
每个优化点需要包含：问题描述 → 优化策略 → 预期收益。
```

#### S - Style（风格）

指定输出的写作或编码风格，可以是某种文体（如"专业报告风格"）、某位专家的写作方式（如"Yann LeCun 的技术博客风格"），或某种体裁（如"知乎技术帖"）。

```
# 示例
请以《MIT Technology Review》的技术深度报道风格撰写，
类似 Veronique Greenwood 的写作方式：精确但不枯燥，
擅长用具体数字和类比降低理解门槛。
```

#### T - Tone（语气）

设定响应的情感质量和态度。Tone 与 Style 的区别在于：Style 控制表达的形式和方法，Tone 控制情感色彩。

| Tone 选项 | 适用场景 |
|---------|---------|
| 正式 / Formal | 商业报告、学术文档 |
| 热情 / Enthusiastic | 营销内容、激励性写作 |
| 务实 / Pragmatic | 技术决策、工程建议 |
| 赋能 / Empowering | 教育内容、个人成长 |
| 轻松 / Conversational | 内部沟通、教程内容 |

#### A - Audience（受众）

明确响应的目标读者，确保内容深度和术语选择恰到好处。

```
# 示例
目标受众：
- 主修材料科学、辅修计算机的大三学生
- 具备基础 Python 能力，但无 ML 生产部署经验
- 需要快速理解概念用于课程项目
```

#### R - Response（响应格式）

定义输出的结构、格式和排版要求。

```
# 示例格式选项
- JSON 格式（适合程序处理）
- Markdown 格式（适合文档）
- 表格（适合对比分析）
- 分步骤编号列表（适合流程指引）
- 幻灯片大纲（适合演示汇报）
```

### 1.3 CO-STAR 完整模板

```markdown
# Context（上下文）
[提供详细的背景信息，包括：
- 当前项目/任务的背景
- 相关技术栈和环境约束
- 已有资源和限制条件
- 过往相关决策或进展]

# Objective（目标）
[清晰描述你希望 AI 完成的具体任务，
聚焦于"做什么"而非"怎么做"]

# Style（风格）
[指定期望的输出风格，如：
- 某位专家的写作风格
- 某种文体风格
- 某种技术文档规范]

# Tone（语气）
[设定响应的情感色彩和态度，如：
- 正式/专业
- 轻松/对话式
- 热情但严谨
- 务实直接]

# Audience（受众）
[描述目标读者：
- 他们的背景知识水平
- 他们的关注点和痛点
- 他们需要从输出中获得什么]

# Response（响应格式）
[明确输出格式，如：
- JSON / Markdown / 表格 / 列表
- 字数限制
- 结构层次要求
- 必须包含/禁止包含的元素]
```

### 1.4 实战案例

**场景**：为一家可持续时尚品牌撰写 Instagram 轮播图文案。

```
# Context
我们是一个主打"零废弃"的可持续时尚品牌，目标客群是25-35岁的都市女性，
她们关注环保但不希望以牺牲审美为代价。本月推广主题是"旧衣新生"——
展示我们的面料再生技术如何将工业废料转化为高端面料。

# Objective
撰写一组 Instagram 轮播图文案，共5张，每张包含：
- 一句吸引注意力的标题
- 2-3行正文
- 一个行动号召（CTA）

# Style
参考 Refinery29 的内容风格：年轻、专业、不说教。
数据驱动但讲故事，有明确的女性视角。

# Tone
热情但不夸张，务实但有温度。
避免过度营销化的话术。

# Audience
25-35岁都市女性，受过高等教育，关注可持续生活，
但对"环保说教"有抵触心理。
她们追求的是"好看的环保"，而非"环保的自我牺牲"。

# Response
5张卡片，每张包含：
- [标题] 加粗
- [正文] 2-3行
- [CTA] 以"→"开头
- 整体风格统一，5张卡片之间有叙事逻辑
```

## 二、CRISPE 框架：专业级提示词模板

### 2.1 框架结构

CRISPE（Context, Role, Insight, Statement, Personality, Experiment）是 CO-STAR 的进阶版，额外增加了 **Insight（洞察）** 和 **Personality（人格）** 两个维度，更适合需要 AI 扮演特定角色并持续产出的场景。

| 组件 | 含义 | 核心问题 |
|------|------|---------|
| **C** Context | 背景上下文 | 任务在什么环境下发生？ |
| **R** Role | 扮演的角色 | AI 应该以什么身份回应？ |
| **I** Insight | 深层洞察 | 需要 AI 了解哪些内部信息/约束？ |
| **S** Statement | 任务声明 | 明确要完成什么？ |
| **P** Personality | 人格特质 | 输出的语气、风格有何要求？ |
| **E** Experiment | 实验/迭代 | 期望哪种程度的创造力或探索性？ |

### 2.2 深度解析

**Insight** 是 CRISPE 区别于其他框架的关键。它允许你向 AI 提供通常不会出现在公开知识中的内部信息——公司战略、用户研究数据、行业内部术语等。这在企业场景中尤为关键，因为通用 AI 模型缺乏你公司的专属知识。

**Experiment** 维度则针对 AI 的创造力进行调控。例如，你可以要求 AI "严格遵循现有风格不偏离"，或"大胆尝试新表达方式并说明理由"。这一维度在需要平衡一致性和创新性的场景中（如品牌内容创作）极为有用。

### 2.3 CRISPE 模板示例

```
# Context
[公司的技术债务现状：3个遗留 monolith 服务需要逐步拆解，
团队共8人，其中2人熟悉微服务架构。新功能开发不能中断，
需要在6个月内完成第一阶段拆分。]

# Role
你是一位有15年经验的后端架构师，专精于微服务拆分和渐进式架构演进。
你曾在 Netflix、Stripe 等公司处理过大规模遗留系统迁移。

# Insight
- 团队预算有限，不能引入过多商业工具
- 技术选型必须兼容现有 Kubernetes 集群（v1.26）
- CTO 的优先级是：稳定性 > 性能 > 开发速度

# Statement
制定一份三阶段微服务拆分计划。
每阶段需要包含：目标 → 拆分策略 → 风险点 → 验收标准。

# Personality
务实、直接、数据驱动。
偏好低风险渐进式方案，反对"大爆炸"式重构。
报告风格偏向技术管理视角（给 CTO 看）。

# Experiment
在每个阶段末尾，提供一个"可选的激进方案"作为对比，
说明在什么条件下可以加速执行。
```

## 三、ICIO 框架：指令-上下文-格式

### 3.1 框架概述

ICIO 框架是一种更简洁的提示词结构，专注于四个核心要素，适合快速迭代和简单任务。它是 CO-STAR 和 CRISPE 的轻量替代方案。

| 组件 | 含义 | 说明 |
|------|------|------|
| **I** Instruction | 核心指令 | 告诉 AI 要做什么 |
| **C** Context | 上下文背景 | 提供必要的背景信息 |
| **I** Input format | 输入格式 | 说明提供数据的方式 |
| **O** Output format | 输出格式 | 指定返回的结构 |

### 3.2 ICIO 模板与案例

**模板：**
```markdown
# Instruction
[明确、具体的任务描述]

# Context
[必要的背景信息]

# Input format
[我将以什么格式提供输入]
（例如：JSON / Markdown 表格 / 纯文本 / 图片）

# Output format
[我期望的输出格式]
（例如：JSON / Markdown / 表格 / 列表）
```

**实战案例**（数据分析）：

```
# Instruction
分析以下用户行为数据，找出流失率最高的使用场景，
并提出三个可立即执行的改善建议。

# Context
我们的 SaaS 产品是一款面向独立设计师的项目管理工具。
当前月度流失率为 18%，高于行业平均的 12%。
目标是将流失率降至 10% 以下。

# Input format
我将以 CSV 格式提供过去90天的用户行为日志，
包含字段：user_id, event_type, timestamp, session_duration, feature_used。

# Output format
1. 数据洞察摘要（200字以内）
2. 流失率Top3场景的详细分析（每项包含数据支撑）
3. 三个可执行改善建议（按优先级排序，包含预期影响估算）
```

## 四、BROKE 框架：目标驱动型提示

### 4.1 框架概述

BROKE 框架由五步构成，特别适合需要明确目标和可衡量成果的业务场景。它强调通过"演化"（Evolve）步骤实现提示词的自我优化。

| 组件 | 含义 | 关键问题 |
|------|------|---------|
| **B** Background | 背景描述 | 任务在什么情境下发生？ |
| **R** Role | 角色定义 | AI 应该是什么身份？ |
| **O** Objective | 目标明确 | 核心目标是什么？ |
| **K** Key Result | 关键成果 | 如何衡量成功？ |
| **E** Evolve | 迭代优化 | 如何改进当前方案？ |

### 4.2 BROKE 的独特价值：Evolve 步骤

Evolve 是 BROKE 最具创新的部分。它要求 AI 在给出初始方案后，主动提出三个维度的改进建议：

1. **逻辑清晰度**：是否有逻辑漏洞或不一致之处？
2. **用例扩展**：方案能否应用于更多场景？
3. **语言风格调整**：输出风格是否需要优化？

这使得 BROKE 不仅是一个提示词模板，更是一个自我改进的机制。

### 4.3 BROKE 模板

```
# Background
[描述任务背景和当前挑战]

# Role
[定义 AI 的角色身份和专业背景]

# Objective
[明确核心目标和预期成果]

# Key Result
[定义可衡量的成功指标]
（例如：转化率提升15%、报告被核心决策者采纳）

# Evolve
请在提供方案后，从以下三个维度各提出一个改进建议：
1. 逻辑清晰度提升
2. 应用场景扩展
3. 表达风格优化
```

## 五、其他结构化框架快速一览

### 5.1 RTF 框架（Role-Task-Format）

最快速的格式化框架，适合需要特定输出格式的场景。

```
# Role：你是一位资深产品经理
# Task：为一个 ToC 的 Markdown 笔记应用设计用户调研问卷
# Format：请以 JSON 格式输出，包含问卷标题、问题列表（每个问题包含类型、题干、选项）
```

### 5.2 APE 框架（Ask-Plan-Execute）

适合多步骤问题：AI 先复述目标，再制定计划，最后执行。

```
请帮我设计一个新产品的发布计划。
步骤：
1. (Ask) 重述产品发布的核心目标
2. (Plan) 列出包含时间线的发布计划
3. (Execute) 详细展开每个阶段的关键动作
```

### 5.3 SOP 框架（Standard Operating Procedure）

适合需要 AI 按照标准化流程执行的任务。

```
请按照以下 SOP（标准操作流程）处理用户反馈：
Step 1: 分类反馈类型（Bug/功能请求/体验投诉/其他）
Step 2: 提取关键信息（用户环境、复现步骤、影响范围）
Step 3: 优先级评估（基于影响范围 × 复现频率）
Step 4: 输出结构化报告
```

### 5.4 TEMPLATE 框架

| 组件 | 含义 |
|------|------|
| **T** Task | 核心任务 |
| **E** Examples | 示范示例 |
| **M** Mentions | 注意事项 |
| **P** Assumptions | 假设条件 |
| **L** Logic | 逻辑要求 |
| **A** Updates | 更新机制 |
| **T** Tone | 语气要求 |
| **E** Evaluation | 评估标准 |

### 5.5 框架对比速查表

| 框架 | 最适合场景 | 复杂度 | 组件数 |
|------|-----------|--------|--------|
| CO-STAR | 竞赛级复杂任务 | 高 | 6 |
| CRISPE | 企业级专业任务 | 很高 | 6 |
| ICIO | 快速迭代简单任务 | 低 | 4 |
| BROKE | 目标驱动型业务任务 | 中 | 5 |
| RTF | 格式严格的技术任务 | 很低 | 3 |
| APE | 多步骤复杂问题 | 中 | 3 |
| SOP | 标准化流程任务 | 低 | N |
| TEMPLATE | 高可控性生产任务 | 高 | 8 |

## 六、链式思考 CoT：推理增强基础

### 6.1 什么是 Chain-of-Thought

Chain-of-Thought（链式思考）是由 Wei et al. (2022) 提出的提示技术，其核心思想是**让 LLM 在给出最终答案前，展示逐步的推理过程**。这看似简单，却对推理能力有显著提升。

### 6.2 CoT 的两种形态

#### Zero-Shot CoT

通过一句引导语激活推理模式，无需示例：

```
问题：小明有12个苹果，送给小红一半，又买了8个。
小明现在有多少个苹果？

让我们一步一步地思考：
```

#### Few-Shot CoT

通过多个包含完整推理链的示例，让模型学习推理模式：

```
示例1：
问题：小张有5个橘子，给了弟弟3个，又收到了4个。小张有几个橘子？
推理：小张有5个橘子。给了弟弟3个，剩下5-3=2个。
      又收到4个，变成2+4=6个。
答案：6个

问题：[你的实际问题]
推理：
```

### 6.3 CoT 的适用边界

CoT 在以下任务中效果显著：

- **算术推理**：多步数学应用题
- **常识推理**：需要生活知识的多跳问题
- **符号推理**：逻辑推导、模式识别

CoT 在以下场景效果有限：

- **简单任务**：一步到位的直接问题（反而增加开销）
- **低资源模型**：小模型的链式推理本身可能出错
- **闭域事实查询**：日期查询、定义查找等

### 6.4 CoT 进阶技巧

**分离推理和答案**：将思考过程与最终答案用不同标记区分：

```
<thinking>
[在此处展开详细推理过程，占用尽可能多的推理 token]
</thinking>

<answer>
[最终答案，仅输出核心结论]
</answer>
```

**强制推理深度**：对于复杂问题，要求 AI"至少推理 N 步"：

```
请在回答前进行至少7步独立的推理推导。
每一步必须：
1. 陈述当前已知信息
2. 说明应用什么规则或关系
3. 推导中间结论
4. 更新已知信息集合
```

## 七、思维树 ToT：复杂问题的树状探索

### 7.1 从线性链到树状结构

Tree of Thoughts（思维树）由 Yao et al. (2023) 提出，解决了 CoT 线性链的局限。在 ToT 中，LLM 在每个推理步骤可以生成多个候选思路，形成树状探索结构。

### 7.2 ToT 的四个核心步骤

1. **思维分解（Thought Decomposition）**：将问题分解为中间思维步骤
2. **思维生成（Thought Generation）**：在每个节点生成多个候选思路
3. **状态评估（State Evaluation）**：评估每个思路的可行性
4. **搜索算法（Search Algorithm）**：使用 BFS 或 DFS 遍历思维树

### 7.3 ToT 模板示例

```
问题：为一家 AI 初创公司制定2026年Q2产品策略

思维阶段1 - 生成选项（每个生成3个）：
请列出3个不同方向的产品策略选项。

[AI 生成3个选项：A/B/C]

思维阶段2 - 评估每个选项：
请从以下三个维度评估每个选项：
- 市场可行性（1-10分）
- 技术可行性（1-10分）
- 竞争优势（1-10分）

[AI 给出评分和理由]

思维阶段3 - 深度探索Top选项：
选择评分最高的选项，展开详细的执行计划。
对于每个子计划，说明：前提条件 → 关键里程碑 → 潜在风险
```

### 7.4 ToT 与 CoT 的选择

| 场景 | 推荐方法 |
|------|---------|
| 单路径可达的推理问题 | CoT（更高效） |
| 需要探索多种可能性的创意问题 | ToT（更全面） |
| 高风险决策（需要验证） | ToT + Self-Consistency |
| 快速问答 | Zero-Shot CoT |

## 八、ReAct：推理+行动框架

### 8.1 什么是 ReAct

ReAct（Reasoning + Acting）由 Yao et al. (2023) 提出，其核心创新是**将推理过程与外部行动交替进行**。与 CoT 纯推理不同，ReAct 让 LLM 能够使用外部工具（如搜索、计算、API 调用），同时在思考中体现工具的使用意图。

### 8.2 ReAct 的循环结构

```
问题 → 思考1 → 行动1 → 观察1 → 思考2 → 行动2 → 观察2 → ... → 最终答案
```

### 8.3 ReAct 模板

```
你是一个研究助手，可以通过搜索来回答问题。
请按以下格式交替进行思考和行动：

问题：[用户的问题]

思考1：[分析问题，确定需要什么信息]
行动1：[执行搜索或查询，如 search[具体查询词]]
观察1：[分析搜索结果]

思考2：[基于观察确定下一步]
行动2：[继续搜索或整理信息]
...

最终答案：[综合所有信息给出完整回答]
```

### 8.4 ReAct 的典型应用场景

- **多跳问答**：需要整合多个信息源的问题（如"某公司CEO在上次财报中提到的Q3目标是什么？"）
- **动态信息查询**：实时新闻、股价、天气等需要最新数据的问题
- **需要验证的事实性问题**：通过多次搜索交叉验证信息

## 九、自洽性 Self-Consistency

### 9.1 原理

Self-Consistency（自洽性）由 Wang et al. (2022) 提出，解决 CoT 中单一推理路径可能出错的问题。其核心思想是**利用 LLM 的随机性，采样多条推理路径，通过多数投票选择最一致的答案**。

### 9.2 为什么 Self-Consistency 有效

LLM 的随机性（由 temperature 参数控制）导致相同提示产生不同输出。如果正确推理路径产生答案 A，错误推理路径产生答案 B/C/D，那么通过多数投票，A 将被选为最终答案。

### 9.3 实战配置

```python
from openai import OpenAI

client = OpenAI()

# 采样多条推理路径
responses = []
for _ in range(5):
    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "system", "content": "你是一个数学推理助手。请在回答时先展示推理过程，再给出最终答案。"},
            {"role": "user", "content": "鸡兔同笼问题：有35个头，94只脚。问鸡和兔各多少只？请逐步推理。"}
        ],
        temperature=0.7,  # 适度随机性
        max_tokens=500
    )
    responses.append(response.choices[0].message.content)

# 提取答案并投票
# （实际实现中需要从响应中解析数字答案）
```

### 9.4 CoT + Self-Consistency 的组合使用

两者结合可以显著提升复杂推理任务的准确率：

```
配置建议：
- temperature: 0.5-0.8（太低无多样性，太高推理链断裂）
- 采样次数: 5-21（奇数次便于投票）
- CoT 引导语："请逐步推理，展示完整的思考过程"
```

## 十、分解类技术：Least-to-Most 与 Plan-and-Solve

### 10.1 Least-to-Most（由简入繁）

Least-to-Most（LtM）通过两阶段分解解决复杂问题：

1. **分解阶段**：将复杂问题拆解为有序的子问题列表
2. **求解阶段**：依次解决每个子问题，用前序答案引导后续求解

LtM 的核心优势在于**泛化能力**：模型能解决比训练样本更复杂的问题，因为分解策略是可迁移的。

```
# LtM 模板

## 分解阶段
请将以下问题分解为3-7个有序的子问题。
每个子问题应该比原问题更简单，且前序子问题的答案将帮助解决后续子问题。

原问题：[你的复杂问题]

子问题列表：
1. ...
2. ...
...

## 求解阶段
请依次解答上述子问题，并利用前序答案引导后续求解。

子问题1：[第一个子问题]
答案1：[...]
```

### 10.2 Plan-and-Solve（规划-执行）

Plan-and-Solve（PaS）强制模型在执行前先制定计划：

1. **规划阶段**：理解问题、提取关键信息、制定行动路线
2. **执行阶段**：按计划逐步计算，推导出最终答案

PaS 特别适合：多约束任务、长多步推理、以及模型容易跳过逻辑步骤的场景。

```
# PaS 模板

请按以下两步回答问题：

## 第一步：制定计划
1. 理解问题的核心要求
2. 提取题目中的关键数据
3. 规划求解路线（需要哪些中间步骤）

## 第二步：执行计划
1. [按计划执行第一步]
2. [执行第二步]
...
3. 最终答案：[结论]
```

### 10.3 LtM vs PaS 对比

| 维度 | Least-to-Most | Plan-and-Solve |
|------|--------------|----------------|
| 核心理念 | 分解为简单子问题依次求解 | 先规划再执行 |
| 透明度 | 中间步骤完全可见 | 计划阶段提供高层路线 |
| 适用场景 | 层次化问题集 | 多约束复杂推理 |
| 成本 | 子问题数量决定 token 量 | 固定两阶段 |

## 十一、Agentic Prompting：自主智能体提示

### 11.1 什么是 Agentic Prompting

Agentic Prompting 超越了单轮交互，是一套让 AI 系统能够**自主分解复杂任务、使用工具、自我修正**的提示技术体系。在 2025-2026 年的推理模型时代，Agentic Prompting 已成为生产级 AI 系统的标配。

### 11.2 核心模式

#### 任务分解模式

```
你是一个自主任务执行助手。对于收到的任务，请：
1. 分析任务的完整边界和约束
2. 将任务分解为独立的子任务
3. 为每个子任务确定：优先级、执行顺序、所需工具
4. 依次执行并记录中间结果
5. 汇总所有子任务成果给出最终输出

如果遇到不确定的情况：
- 优先使用工具验证
- 记录假设条件
- 在最终输出中说明不确定性
```

#### 自我验证模式

```
在给出最终答案前，请进行自我验证：
1. 回顾原始问题：我的回答是否真正解决了问题？
2. 检查逻辑链：每一步推理是否有漏洞？
3. 压力测试：如果前提条件改变，答案还成立吗？
4. 反例搜索：能否找到一个反例推翻我的结论？

如果发现错误，回到推理阶段重新推导。
```

#### 批评-修订循环

```
请按照以下循环完成写作任务：

循环 N=1 到 3：
  阶段1（起草）：基于当前上下文撰写内容
  阶段2（批评）：从以下维度评估草稿：
    - 逻辑连贯性
    - 目标受众的接受度
    - 表达清晰度
    - 是否偏离核心主题
  阶段3（修订）：根据批评意见修订草稿

最终输出第3次修订后的版本。
```

### 11.3 工具优先思维

在 Agentic 场景中，核心原则是**优先使用确定性工具，而非生成性输出**：

```
# ❌ 低效方式
"请列出北京市所有三甲医院及其地址"

# ✅ 工具优先
"请使用搜索引擎查询北京市三甲医院名单，
然后从搜索结果中提取医院名称和地址，
整理为表格格式"
```

## 十三、RAG 系统中的提示词工程

### 13.1 RAG 提示词的特殊性

在 RAG（检索增强生成）系统中，提示词工程面临独特的挑战：系统提示词必须同时管理持久规则、动态检索内容、对话历史和输出格式。

### 13.2 RAG 双层提示结构

#### System Prompt（系统提示）

```markdown
你是[公司名]的智能客服助手，名为[助手名]。

## 核心原则
- 仅使用提供的检索内容回答问题
- 如果检索内容不足以回答，请明确说明"基于我检索到的信息..."
- 永远不要编造不在检索内容中的事实
- 保持专业、友好、简洁的回复风格

## 拒绝模式
以下问题必须拒绝回答：
- 涉及公司商业机密的问题
- 对竞品的直接评价
- 法律、医疗、金融领域的具体建议

## 格式要求
- 回答控制在200字以内
- 使用 Markdown 格式化
- 在引用信息后标注来源
```

#### User Prompt（用户提示）

```markdown
## 检索内容
---
{retrieved_context}
---

## 用户问题
{user_question}

## 回答要求
请基于上述检索内容回答用户问题。
如果检索内容不足以完整回答，请说明哪些部分可以回答、哪些需要补充信息。
引用信息时使用以下格式：[来源编号]
```

### 13.3 Advanced RAG 提示词模板

#### Query Expansion（查询扩展）

```
你是一个查询优化助手。请将用户的问题扩展为3个不同角度的表述，
以帮助从不同维度检索相关信息。

原始问题：[用户问题]

扩展后的查询（请保持核心语义不变）：
1. [同义词替换角度]
2. [上位概念角度]
3. [下位具体化角度]
```

#### HyDE（Hypothetical Document Embeddings）

```
请先假设一个可能回答问题的文档应该包含什么内容，
然后基于这个假设描述来检索相关资料。

问题：[用户问题]
假设的答案文档应该包含：[生成2-3句话的假设性摘要]
```

#### Multi-Query（多查询路由）

```
请将以下问题改写为3个不同的查询表述，
覆盖问题的不同方面和可能的表达方式。

原始问题：[你的问题]

改写1（更具体）：
改写2（更抽象）：
改写3（不同表述方式）：
```

## 十四、生产级提示词最佳实践

### 14.1 提示词分层架构

在生产级应用中，提示词应遵循分层设计：

| 层级 | 内容 | 变更频率 |
|------|------|---------|
| **System Layer** | 角色定义、安全规则、格式规范 | 低（长期稳定） |
| **Developer Layer** | 任务模板、Few-shot 示例库 | 中（随需求调整） |
| **User Layer** | 实际用户输入 | 高（每次交互变化） |

### 14.2 通用最佳实践清单

#### ✅ 有效实践

1. **显式指定输出格式**：与其说"格式清晰"，不如说"使用 Markdown H2 标题分节"
2. **使用积极指令**：说"你应该做X"而非"不要做Y"
3. **分离 Style 和 Tone**：CO-STAR 框架的核心洞察
4. **提供具体约束**：数量限制、字数限制、禁止提及的元素
5. **使用 XML 标签分隔内容**：便于模型识别不同部分
6. **渐进式提供信息**：先问关键问题，再根据回答补充细节
7. **善用 Few-shot 示例**：即使是1-2个高质量示例也能显著提升效果

#### ❌ 应避免的做法

1. **模糊的目标描述**："帮我写好一点"无意义
2. **过度约束**：过多的禁止条款会压缩有效输出空间
3. **隐含假设**：期望模型"自然会懂"而不明确说明
4. **矛盾指令**：同时说"详细展开"和"简洁明了"
5. **缺乏边界定义**：不说清可以做什么、不可以做什么

### 14.3 提示词版本管理

```
提示词 V1.2.3
---
变更记录：
V1.2.3（2026-04-10）：增加"格式要求"章节，减少幻觉
V1.2.2（2026-04-05）：优化Tone为"务实但不冷漠"
V1.2.1（2026-04-02）：修正受众描述，聚焦25-35岁
V1.2.0（2026-04-01）：从ICIO迁移到CO-STAR框架
---
测试用例：
- [测试用例1描述] → 期望输出：[...]
- [测试用例2描述] → 期望输出：[...]
```

### 14.4 成本优化策略

1. **模型分层使用**：简单任务用小模型（GPT-4o-mini/Claude Haiku），复杂任务用大模型
2. **上下文压缩**：在长对话中使用摘要而非完整历史
3. **结构化输出**：用 JSON Schema 约束输出，减少 token 浪费
4. **增量提示**：仅传递变化的部分，而非完整上下文

## 十五、提示词评估与迭代优化

### 15.1 评估维度

| 维度 | 衡量指标 | 评估方法 |
|------|---------|---------|
| **相关性** | 输出是否切题 | 人工抽检 / 自动评分 |
| **准确性** | 事实性错误数量 | 对比标准答案 |
| **格式合规性** | 是否按要求输出 | Schema 验证 |
| **一致性** | 同类问题的一致程度 | 多次采样方差分析 |
| **有害性** | 是否包含有害内容 | 安全分类器检测 |
| **成本效率** | Token 消耗与效果比 | 成本追踪 |

### 15.2 迭代优化流程

```
输入 → 基准测试 → 问题识别 → 提示词修改 → A/B测试 → 部署 → 监控 → 反馈 → 迭代
```

### 15.3 常见问题与解决方案

| 问题 | 表现 | 解决方案 |
|------|------|---------|
| 幻觉 | 编造不存在的事实 | 增加上下文约束 + 要求引用来源 |
| 格式偏差 | 输出结构不稳定 | 强化 Response 描述 + Few-shot 示例 |
| 过度冗长 | 输出超过字数限制 | 明确字数上限 + 分点要求 |
| 角色偏离 | 回复风格不一致 | 强化 Style/Tone 描述 + 增加边界 |
| 推理错误 | 数学/逻辑推导错误 | 引入 CoT + 要求中间步骤验证 |
